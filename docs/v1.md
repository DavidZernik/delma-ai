# Delma V1 — what we're building, why, and how it works

This is the canonical V1 reference for Delma.

Read this once and you should understand the product shape clearly:
what Delma is, what Claude Code is responsible for, why the diagrams
exist, what gets stored, how updates happen, and what the local system
is actually doing under the hood.

Plain English where it can be. Technical where it has to be. No code.

---

## 1. What Delma is, in one sentence

> **Delma is a local visual memory sidecar for Claude Code that keeps a set of beautiful, versioned Mermaid diagrams plus a managed `CLAUDE.md` in sync with your project.**

The important word is **sidecar**.

Delma is not trying to replace Claude Code. Claude Code is still the
main coding surface, the main agent, and the thing doing the work.
Delma exists beside it to make project understanding persistent,
visual, editable, and reusable across sessions.

---

## 2. The strategic bet

The bet behind Delma is that coding agents are getting strong enough
that the bottleneck is no longer raw code generation. The bottleneck is
**working memory**.

Developers do not just need a chat transcript. They need a durable,
editable system map that stays current as the project evolves. They
need something they can glance at in 10 seconds and feel oriented
again. They need something the agent can read too.

Delma is built on three connected bets:

```mermaid
flowchart TB
  bet1["🎯 Bet 1 — Claude Code should stay the main worker<br/><br/><b>The claim:</b> The coding loop already lives in Claude Code. Asking the user to switch into a separate pseudo-chat product creates friction, duplicates behavior, and weakens the thing they already trust.<br/><br/><b>What we ship:</b> Delma as a local sidecar. Claude Code keeps doing the work. Delma exposes tools Claude can call while it works, and Delma owns the memory, diagrams, and project context surfaces.<br/><br/><b>Why it wins:</b> We inherit Claude Code's strengths instead of competing with them. Delma gets to be opinionated about memory and diagrams without re-solving chat."] 

  bet2["🎯 Bet 2 — Project memory should be diagram-first, not document-first<br/><br/><b>The claim:</b> A developer trying to understand a living system wants a visual map before they want prose. A great diagram makes the next question obvious. Raw markdown notes do not.<br/><br/><b>What we ship:</b> A tabbed set of Mermaid views like Codebase, Org, Data Flows, Automations, and Current Work. Each tab answers a distinct question and stays versioned over time.<br/><br/><b>Why it wins:</b> Constrained views stay readable. The product feels useful immediately, not after wading through notes."] 

  bet3["🎯 Bet 3 — Memory only becomes trusted if the user and the agent can both edit it<br/><br/><b>The claim:</b> Invisible background memory feels magical right up until it is wrong. The diagrams, notes, history, and `CLAUDE.md` stop drifting apart."]
```

If those three bets are right, Delma becomes the durable project
understanding layer around Claude Code. Claude Code is the worker.
Delma is the living map.

## 4. The core user journey

Here is the V1 loop we are building around your real workflow.

```mermaid
flowchart TB
  open["📂 You open your project in Delma<br/><br/><b>What happens:</b> Delma initializes `.delma/`, loads or creates the workspace, loads history and memory files, and composes `CLAUDE.md` if needed.<br/><br/><b>Why it matters:</b> Delma becomes grounded in the same repo Claude Code is using."] 

  runClaude["💬 You work in Claude Code<br/><br/><b>What happens:</b> Claude Code stays the main surface for coding, planning, and edits. Delma is open beside it as the visual memory workspace.<br/><br/><b>Why it matters:</b> No duplicate chat experience. No wrapper tax."] 

  callDelma["🔧 Claude Code calls Delma tools while working<br/><br/><b>What happens:</b> Claude reads the current Delma state, updates a view, appends a note, or recomposes `CLAUDE.md` after learning something useful.<br/><br/><b>Why it matters:</b> Memory becomes part of the work itself instead of a separate cleanup task."] 

  update["🗺️ Delma updates diagrams and memory<br/><br/><b>What happens:</b> The relevant Mermaid tab changes, a snapshot is written to `.delma/history/`, and `CLAUDE.md` is rebuilt from the same source of truth.<br/><br/><b>Why it matters:</b> The diagram, notes, and agent-readable memory stay coherent."] 

  return["↩️ You come back tomorrow and still understand the project<br/><br/><b>What happens:</b> Delma shows the same tabs, the same history, and the latest `CLAUDE.md`. The success condition is simple:
```

you feel less blind when you start, and less forgetful when you come

## 5. The five diagram tabs

Delma V1 does not revolve around one giant Mermaid.

It revolves around a **set of views**, each with a different job.
That is how the diagrams stay beautiful instead of turning into a
hairball.

```mermaid
flowchart TB
  codebase["🧱 Codebase<br/><br/><b>Question it answers:</b> What are the important app surfaces, services, modules, or files, and how do they relate?<br/><br/><b>Typical content:</b> frontend, backend, scripts, APIs, data stores, major runtime boundaries."] 

  org["👥 Org<br/><br/><b>Question it answers:</b> Who owns what, who decides what, and what human context matters?<br/><br/><b>Typical content:</b> stakeholders, owners, approval paths, tribal knowledge, risky boundaries."] 

  flows["🔄 Data Flows<br/><br/><b>Question it answers:</b> How does information move through the system?<br/><br/><b>Typical content:</b> APIs, SFMC data extensions, imports, exports, journeys, transformations."] 

  automations["⏱️ Automations<br/><br/><b>Question it answers:</b> What runs on a schedule or trigger, and what depends on it?<br/><br/><b>Typical content:</b> scheduled jobs, scripts, queries, journeys, alerts, failure points."] 

  current["🎯 Current Work<br/><br/><b>Question it answers:</b> What matters for the task I am doing right now?<br/><br/><b>Typical content:</b> touched files, current risk, immediate dependencies, intended outcome."] 

  workspace["🗂️ Delma Workspace"] --> codebase
  workspace --> org
  workspace --> flows
  workspace --> automations
  workspace --> current

  classDef rich font-size:11px,text-align:left
  class workspace,codebase,org,flows,automations,current rich
  style workspace fill:#f6ead6,stroke:#a0611a,stroke-width:3px
```

The product rule is that every tab needs a clear reason to exist.

If a tab does not answer a distinct question, it should not be a tab.
If a tab gets too broad to stay readable, it should split.

## 6. What Delma actually stores

Delma is local-first. Its value comes from keeping project memory near
the repo, visible on disk, and easy for both humans and agents to use.

```mermaid
flowchart TB
  workspace["📄 `.delma/workspace.json`<br/><br/><b>What it is:</b> The source of truth for Delma's diagram workspace.<br/><br/><b>What's inside:</b> project name, the list of view tabs, each view's title, description, summary, Mermaid body, and update timestamp.<br/><br/><b>Why it exists:</b> One structured file is easier to version, inspect, diff, and edit than a scattered set of opaque UI-only state."] 

  history["🕰️ `.delma/history/`<br/><br/><b>What it is:</b> Snapshot files written every time the workspace is saved.<br/><br/><b>What's inside:</b> the full workspace state, a timestamp, and a save reason.<br/><br/><b>Why it exists:</b> Versioning is a product feature, not a debug feature. Users need to know what changed and when."] 

  memory["📝 `.delma/*.md` memory files<br/><br/><b>What they are:</b> Supporting markdown notes like `environment.md`, `logic.md`, `people.md`, and `session-log.md`.<br/><br/><b>Why they exist:</b> Not every useful thing belongs as a diagram edge. These files hold supporting reference context and durable notes."] 

  graph["🧩 `.delma/graph.json`<br/><br/><b>What it is:</b> A placeholder canonical graph store for the next phase of Delma.<br/><br/><b>Why it exists now:</b> The architecture is being prepared for graph-backed view generation rather than raw text-only Mermaid editing forever."] 

  claude["📘 `CLAUDE.md` at project root<br/><br/><b>What it is:</b> The agent-readable memory surface that Claude Code can read naturally at session start.<br/><br/><b>How it is made:</b> Composed from the workspace views plus the memory markdown files.<br/><br/><b>Why it exists:</b> Delma improves Claude Code by feeding it better context, not by replacing it."] 

  workspace --> history
  workspace --> claude
  memory --> claude
  graph -.-> workspace

  classDef rich font-size:11px,text-align:left
  class workspace,history,memory,graph,claude rich
  style claude fill:#e7eef8,stroke:#2d5ea6,stroke-width:3px
```

The important principle is that the diagrams and `CLAUDE.md` come from
the same memory system. They should not drift into separate truths.

## 7. How Claude Code talks to Delma

Delma V1 is built around MCP because Claude Code already knows how to
use MCP servers. That makes the architecture cleaner and more durable
than trying to wrap the Claude CLI in a fake chat shell.

```mermaid
flowchart LR
  claude["🤖 Claude Code"] --> mcp["🧰 Delma MCP server (stdio)"]
  mcp --> open["open_project"]
  mcp --> state["get_delma_state"]
  mcp --> list["list_diagram_views / get_diagram_view"]
  mcp --> save["save_diagram_view"]
  mcp --> note["append_memory_note"]
  mcp --> compose["compose_claude_md"]
  mcp --> history["list_history"]
  mcp --> files["Local `.delma/` files"]

  classDef rich font-size:11px,text-align:left
  class claude,mcp,open,state,list,save,note,compose,history,files rich
  style mcp fill:#f6ead6,stroke:#a0611a,stroke-width:3px
```

The system is intentionally simple.

Claude Code does not need a giant Delma API surface in V1. It needs a
small set of reliable operations that map directly to project memory
tasks.

## 8. The MCP tool contract

Each Delma MCP tool serves a clear user-facing job.

```mermaid
flowchart TB
  open["📂 `open_project`<br/><br/><b>Purpose:</b> Point Delma at the active repo and initialize `.delma/` if needed.<br/><br/><b>Why it matters:</b> Gives Claude Code a clean way to ground Delma in the same project before reading or writing memory."] 

  state["📊 `get_delma_state`<br/><br/><b>Purpose:</b> Return the current workspace, graph, memory files, and history for the project.<br/><br/><b>Why it matters:</b> Lets Claude understand what Delma already knows before updating anything."] 

  list["🗂️ `list_diagram_views` + `get_diagram_view`<br/><br/><b>Purpose:</b> Discover which diagram tabs exist and inspect one in detail.<br/><br/><b>Why it matters:</b> Lets Claude pick the right surface to update instead of dumping everything into one place."] 

  save["✍️ `save_diagram_view`<br/><br/><b>Purpose:</b> Update a tab's title, description, summary, or Mermaid body, then snapshot the workspace and rebuild `CLAUDE.md`.<br/><br/><b>Why it matters:</b> This is the core “Claude learned something important, preserve it” action."] 

  note["📝 `append_memory_note`<br/><br/><b>Purpose:</b> Add supporting prose to one of the markdown memory files.<br/><br/><b>Why it matters:</b> Some context belongs in notes instead of diagrams, especially human constraints and explanatory detail."] 

  compose["📘 `compose_claude_md`<br/><br/><b>Purpose:</b> Regenerate `CLAUDE.md` from Delma state on demand.<br/><br/><b>Why it matters:</b> Keeps Claude's future context synchronized with the latest diagram workspace."] 

  history["🕰️ `list_history`<br/><br/><b>Purpose:</b> Show what snapshots exist for the project.<br/><br/><b>Why it matters:</b> Gives Claude and the user a way to reason about change over time."] 

  open --> state --> list --> save --> compose --> history
  list --> note

  classDef rich font-size:11px,text-align:left
  class open,state,list,save,note,compose,history rich
```

The central action is `save_diagram_view`.

That is the bridge between “Claude learned something” and “the project
now has better memory.”

## 10. The update pipeline

In V1, Delma supports both manual and agent-driven updates.

```mermaid
flowchart TB
  learn["🧠 Something is learned<br/><br/><b>Examples:</b> Claude discovers an ownership boundary, you realize an automation depends on a hidden script, or a current task touches more systems than expected."] 

  choose["🧭 Choose the right memory surface<br/><br/><b>Question:</b> Does this belong in a diagram tab, a markdown memory note, or both?<br/><br/><b>Reason:</b> Not everything should become another Mermaid edge."] 

  saveView["🗺️ Update a diagram tab<br/><br/><b>Action:</b> Save a revised Mermaid view through Delma."] 

  append["📝 Append a note<br/><br/><b>Action:</b> Add supporting explanation to `environment.md`, `logic.md`, `people.md`, or `session-log.md`."] 

  snapshot["🕰️ Write a snapshot<br/><br/><b>Action:</b> Delma stores a versioned copy of the workspace in `.delma/history/`."] 

  compose["📘 Recompose `CLAUDE.md`<br/><br/><b>Action:</b> Delma regenerates the agent-readable memory file from the latest state."] 

  future["↩️ Future Claude sessions start smarter<br/><br/><b>Effect:</b> The next session inherits better context without depending on your own memory or transcript archaeology."] 

  learn --> choose
  choose --> saveView
  choose --> append
  saveView --> snapshot --> compose --> future
  append --> compose

  classDef rich font-size:11px,text-align:left
  class learn,choose,saveView,append,snapshot,compose,future rich
  style compose fill:#e7eef8,stroke:#2d5ea6,stroke-width:3px
```

The key design choice is that `CLAUDE.md` is not hand-maintained.

It is the downstream artifact of Delma’s memory system, not a parallel
thing the user has to babysit.

## 11. Why versioning matters

Versioning is not a nice-to-have for Delma. It is one of the reasons
the product is worth using.

```mermaid
flowchart LR
  noHistory["Without history<br/><br/><b>The problem:</b> The diagram quietly changes over time. Nobody knows what moved, when it moved, or why. Wrong updates feel destructive. Trust erodes fast."] 

  history["With history<br/><br/><b>The behavior:</b> Every save produces a timestamped snapshot. The user can inspect the timeline, compare states, and recover if a change was wrong."] 

  trust["Result: trustable memory<br/><br/><b>Why it matters:</b> Delma can become part of the workflow only if it feels reversible. A memory system you can't inspect is a memory system you eventually avoid."] 

  noHistory --> trust
  history --> trust

  classDef rich font-size:11px,text-align:left
  class noHistory,history,trust rich
  style history fill:#e4f2ec,stroke:#1e5c4a,stroke-width:3px
```

The longer the project lives, the more valuable that timeline becomes.

## 15. The success metric

The most important V1 question is:

> **Do you actually prefer working with Claude Code + Delma over Claude Code alone?**

Everything else is downstream of that.

The practical signals are:

1. You keep Delma open while working.
2. Claude Code starts using Delma memory naturally.
3. The diagrams stay current enough to be useful.
4. `CLAUDE.md` gets better instead of rotting.
5. Coming back to a project feels much less disorienting.

If Delma achieves that, V1 is working.

## 16. The TL;DR

1. **Claude Code remains the main worker.** Delma does not replace it.
2. **Delma is the local visual memory sidecar.** It owns diagrams, notes, history, and `CLAUDE.md`.
3. **The core artifact is a tabbed Mermaid workspace, not one giant graph.**
4. **Claude Code updates Delma through MCP tools while it works.**
5. **Every save is versioned.** History is part of the product.
6. **`CLAUDE.md` is generated from Delma state, not maintained by hand.**
7. **The whole point is continuity.** You and Claude should both start the next session smarter.

The product is not “another coding agent.”

The product is **persistent project understanding, living beside
Claude Code, in a form both you and the agent can use.**
